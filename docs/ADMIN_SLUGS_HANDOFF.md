# Slug-поля в админ-панели — справочник для фронтенд-разработчика

> Дата: 2026-03-15  
> Бэкенд-API: `https://trihoback.mediann.dev/api/v1`

---

## Общий принцип

Каждая публично доступная сущность (врач, город, мероприятие, статья, тема статей, документ организации) имеет поле **`slug`** — человекочитаемый идентификатор для URL.

- **Автогенерация** — если админ **не передаёт** slug при создании, бэкенд генерирует его автоматически из заголовка/имени (с транслитерацией кириллицы).
- **Ручной ввод** — админ может задать свой slug при создании или обновлении. Бэкенд его примет «как есть».
- **Уникальность** — бэкенд гарантирует уникальность. При коллизии автоматически добавляется суффикс `-2`, `-3` и т.д.

### Рекомендация для UI

В формах создания/редактирования:
1. Отображать текстовое поле «Slug (URL)» как **необязательное**.
2. Рядом с полем — подсказка: _«Оставьте пустым для автоматической генерации из названия»_.
3. После сохранения — показывать сгенерированный slug в ответе (поле `slug` в response).
4. При редактировании предзаполнять поле текущим значением slug.

## Сущности

### 1. Врачи (DoctorProfile)

| Действие | Endpoint | Поле slug |
|---|---|---|
| Создание врача вручную | `POST /admin/doctors` | Slug генерируется автоматически из `{last_name} {first_name}`. Админ **не может** задать slug вручную при создании. |
| Просмотр врача | `GET /admin/doctors/{profile_id}` | Поле `slug` возвращается в ответе `DoctorDetailResponse`. |
| Модерация (approve) | `POST /admin/doctors/{profile_id}/moderate` | При одобрении slug генерируется автоматически (если ещё не задан). |
| Импорт из Excel | `POST /admin/doctors/import` | Slug генерируется автоматически для каждого импортированного врача. |

**Пример ответа** (`DoctorDetailResponse`):
```json
{
  "id": "aaaaaaaa-...",
  "slug": "petrov-ivan",
  "first_name": "Иван",
  "last_name": "Петров",
  ...
}
```

> Slug врача доступен только для чтения в админке (задаётся автоматически).

---

### 2. Города (City)

| Действие | Endpoint | Поле slug |
|---|---|---|
| Создание города | `POST /admin/cities` | `slug?: string` (необязательно). Если не передан — генерируется из `name`. |
| Редактирование | `PUT /admin/cities/{id}` | `slug?: string`. Если передан — обновляется. Если изменяется `name` и slug не передан — slug перегенерируется. |
| Список городов | `GET /admin/cities` | Поле `slug` в каждом элементе. |

**Пример запроса** (создание):
```json
{
  "name": "Санкт-Петербург",
  "slug": "spb",
  "sort_order": 2
}
```

**Пример ответа:**
```json
{
  "id": "...",
  "name": "Санкт-Петербург",
  "slug": "spb",
  "sort_order": 2,
  "is_active": true,
  "doctors_count": 0
}
```

---

### 3. Мероприятия (Event)

Мероприятия используют `multipart/form-data` (из-за загрузки обложки).

| Действие | Endpoint | Поле slug |
|---|---|---|
| Создание | `POST /admin/events` | Поле формы `slug` (text, необязательно). Если не передано — генерируется из `title`. |
| Редактирование | `PATCH /admin/events/{id}` | Поле формы `slug` (text, необязательно). Если передано — обновляется. |
| Просмотр | `GET /admin/events/{id}` | Поле `slug` в ответе. |
| Список | `GET /admin/events` | Поле `slug` в каждом элементе. |

**Пример запроса** (FormData):
```
title: Конференция 2026
slug: conference-2026
event_date: 2026-06-15T10:00:00Z
status: upcoming
```

**Пример ответа** (`EventCreatedResponse`):
```json
{
  "id": "...",
  "title": "Конференция 2026",
  "slug": "conference-2026",
  "status": "upcoming",
  "created_at": "..."
}
```

## Валидация slug на фронте (рекомендация)

Если админ вводит slug вручную, рекомендуется:
- Разрешать символы: `a-z`, `0-9`, `-` (дефис)
- Удалять пробелы и спецсимволы
- Приводить к lowercase
- Максимальная длина: 255 символов (500 для Event и Article)

Бэкенд принимает slug «как есть» — фронт может делать клиентскую валидацию для удобства.
