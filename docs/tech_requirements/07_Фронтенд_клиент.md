### 1.3 SEO-стратегия

### 1.4 generateMetadata() — шаблоны

Все динамические публичные страницы используют `generateMetadata()` из Next.js App Router. Каждая функция делает серверный запрос к API для получения данных и формирует объект метаданных.

**`/doctors/[slug]`:**
```typescript
export async function generateMetadata({ params }: { params: { slug: string } }): Promise<Metadata> {
  const doctor = await getDoctorBySlug(params.slug); // GET /api/v1/doctors/{slug}
  return {
    title: `${doctor.last_name} ${doctor.first_name} — врач-трихолог в ${doctor.city} | РОТА`,
    description: doctor.bio ?? `Профиль врача-трихолога ${doctor.last_name} ${doctor.first_name}`,
    openGraph: {
      title: `${doctor.last_name} ${doctor.first_name} | РОТА`,
      description: doctor.bio,
      url: `${BASE_URL}/doctors/${params.slug}`,
      type: 'profile',
      images: doctor.photo_url ? [{ url: doctor.photo_url, alt: `${doctor.last_name} ${doctor.first_name} — врач-трихолог` }] : [],
    },
    twitter: { card: 'summary_large_image' },
    alternates: { canonical: `${BASE_URL}/doctors/${params.slug}` },
  };
}
```

**`/events/[slug]`:**
```typescript
export async function generateMetadata({ params }: { params: { slug: string } }): Promise<Metadata> {
```

### 1.5 generateStaticParams() — статическая генерация

Для slug-страниц используется `generateStaticParams()` для ISR/SSG — предварительная генерация путей при сборке:

```typescript
// /doctors/[slug]/page.tsx
export async function generateStaticParams() {
  const doctors = await getDoctorsList({ limit: 1000 }); // GET /api/v1/doctors?limit=1000
  return doctors.data.map((d) => ({ slug: d.slug })).filter(Boolean);
}

// /events/[slug]/page.tsx
export async function generateStaticParams() {
  const events = await getEventsList({ limit: 1000 }); // GET /api/v1/events?limit=1000
  return events.data.map((e) => ({ slug: e.slug }));
}

// /articles/[slug]/page.tsx
export async function generateStaticParams() {
  const articles = await getArticlesList({ limit: 1000 }); // GET /api/v1/articles?limit=1000
  return articles.data.map((a) => ({ slug: a.slug }));
}

// /cities/[slug]/page.tsx
export async function generateStaticParams() {
  const cities = await getCitiesList(); // GET /api/v1/cities
  return cities.data.map((c) => ({ slug: c.slug }));
}
```

Все slug-страницы используют `revalidate` для инкрементальной регенерации:
- `/doctors/[slug]` — `revalidate: 3600` (1 час)
- `/events/[slug]` — `revalidate: 1800` (30 мин)
- `/articles/[slug]` — `revalidate: 3600` (1 час)
- `/cities/[slug]` — `revalidate: 86400` (24 часа)

### 1.6 Structured Data (JSON-LD)

JSON-LD вставляется в `<head>` через компонент `<script type="application/ld+json">`. Используется библиотека `schema-dts` (TypeScript-типы) или ручные объекты.

**`/doctors/[slug]` — схема `Person`:**
```json
{
  "@context": "https://schema.org",
  "@type": "Person",
  "name": "Петров Иван Сергеевич",
  "jobTitle": "Врач-трихолог",
  "worksFor": { "@type": "Organization", "name": "Клиника А" },
  "address": { "@type": "PostalAddress", "addressLocality": "Москва" },
  "image": "https://...",
  "url": "https://[DOMAIN]/doctors/petrov-ivan-sergeevich"
}
```

**`/events/[slug]` — схема `Event`:**
```json
{
  "@context": "https://schema.org",
  "@type": "Event",
  "name": "Конференция 2026",
  "startDate": "2026-05-15T10:00:00+03:00",
  "endDate": "2026-05-15T18:00:00+03:00",
  "location": { "@type": "Place", "name": "Москва", "address": "..." },
  "organizer": { "@type": "Organization", "name": "РОТА", "url": "https://[DOMAIN]" },
  "url": "https://[DOMAIN]/events/konferenciya-2026",
  "image": "https://..."
}
```

**`/articles/[slug]` — схема `Article`:**
```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "Название статьи",
  "datePublished": "2026-01-15T10:00:00Z",
  "dateModified": "2026-01-20T10:00:00Z",
  "author": { "@type": "Organization", "name": "РОТА" },
  "image": "https://...",
  "url": "https://[DOMAIN]/articles/nazvanie-stati"
}
```

**Главная `/` — `Organization` + `WebSite`:**
```json
{
  "@context": "https://schema.org",
  "@type": "Organization",
  "name": "Российское Общество Трихологов Ассоциация",
  "url": "https://[DOMAIN]",
  "logo": "https://[DOMAIN]/logo.png"
}
```

**Хлебные крошки — схема `BreadcrumbList`** (добавляется на все вложенные страницы совместно с `<Breadcrumb>` компонентом):
```json
{
  "@context": "https://schema.org",
  "@type": "BreadcrumbList",
  "itemListElement": [
    { "@type": "ListItem", "position": 1, "name": "Главная", "item": "https://[DOMAIN]/" },
    { "@type": "ListItem", "position": 2, "name": "Врачи", "item": "https://[DOMAIN]/doctors" },
    { "@type": "ListItem", "position": 3, "name": "Петров Иван", "item": "https://[DOMAIN]/doctors/petrov-ivan" }
  ]
}
```

### 1.7 Breadcrumb (UI-компонент)

Используется компонент `<Breadcrumb>` из `shadcn/ui`. Отображается на всех вложенных страницах.

**Маппинг хлебных крошек по страницам:**

| Страница | Путь хлебных крошек |
|---|---|
| `/doctors/[slug]` | Главная → Врачи → {ФИО} |
| `/cities/[slug]` | Главная → Города → {city.name} |
| `/events/[slug]` | Главная → Мероприятия → {event.title} |
| `/articles/[slug]` | Главная → Статьи → {article.title} |

Компонент рендерится как Server Component, данные для крошек передаются через props из page-компонента. JSON-LD `BreadcrumbList` рендерится рядом с UI-компонентом.

### 1.8 Canonical URL — правила

| Ситуация | Canonical |
|---|---|
| Статичная страница без query-params | Текущий URL без параметров |
| `/doctors?city_slug=moskva` | `/cities/moskva` (canonical = страница города) |
| `/doctors?city_id={uuid}` | `/cities/{slug}` (resolve slug → canonical) |
| Пагинация (`?offset=20`) | Первая страница (без `offset`) |
| Явный `canonical_url` в `seo` | Значение из `seo.canonical_url` |
| Slug-страницы | `${BASE_URL}/{entity}/{slug}` |

Canonical вставляется через `alternates.canonical` в `generateMetadata()`.

### 1.9 next/image и alt-атрибуты

**Правило:** Все изображения — строго через `<Image>` из `next/image`. Запрещено использование нативного `<img>` для контентных изображений.

**Обязательные атрибуты:** `src`, `width`, `height`, `alt`.

**Alt-шаблоны:**

| Контекст | Alt |
|---|---|
| Фото врача | `{last_name} {first_name} — врач-трихолог` |
| Обложка мероприятия | `{event.title}` |
| Обложка статьи | `{article.title}` |
| Логотип | `РОТА — Российское Общество Трихологов` |
| Фото диплома | `Диплом {last_name} {first_name}` |
| Декоративные (фоны, иконки) | `alt=""` (пустой, скрыт от screen readers) |

Для изображений из S3 указывается домен в `next.config.js` (поле `images.remotePatterns`).

## 4. Роутинг и защита маршрутов

### 4.1 Middleware (Next.js)

```
middleware.ts:
1. Публичные маршруты (/, /doctors, /cities, /events, /articles, /documents, /login,
   /register, /forgot-password, /reset-password, /confirm-email)
   → пропускать без проверки

2. /onboarding/* — требуется авторизация
   → Если нет токена → redirect /login?redirect={current_url}

3. /cabinet/* — требуется авторизация + завершённый онбординг
   → Если нет токена → redirect /login?redirect={current_url}
   → Если онбординг не завершён → redirect на нужный шаг

4. /login, /register — если уже авторизован → redirect /cabinet
```

### 4.2 Логика редиректов по статусу онбординга

При каждом входе в `/cabinet` или `/onboarding` фронтенд вызывает `GET /api/v1/onboarding/status` и определяет `next_step`:

| `next_step`          | Redirect                     |
|----------------------|------------------------------|
| `verify_email`       | `/register` + сообщение «Проверьте почту» |
| `choose_role`        | `/onboarding/role`           |
| `fill_profile`       | `/onboarding/profile`        |
| `upload_documents`   | `/onboarding/profile` (секция документов) |
| `submit`             | `/onboarding/profile` (кнопка «Отправить заявку») |
| `await_moderation`   | `/onboarding/pending`        |
| `pay_entry_fee`      | `/cabinet/payments` (с CTA)  |
| `completed`          | `/cabinet` (без редиректа)   |

### 4.3 Поведение при истёкшей сессии

1. Axios interceptor перехватывает 401 на любой запрос
2. Пытается обновить токен: `POST /api/v1/auth/refresh`
3. Если refresh возвращает 401 → очистить токен из памяти → `redirect /login?redirect={encodeURIComponent(currentUrl)}`
4. Toast: «Сессия истекла. Войдите заново»
5. После успешного логина → redirect на сохранённый URL из `?redirect`

| Описание | `tariff.description` | Если не null |
| Условия | `tariff.conditions` | Если не null |
| Преимущества | `tariff.benefits[]` — список с иконкой ✓ | Если массив не пустой |
| Доп. детали | `tariff.details` | Если не null |
| Обычная цена | `tariff.price` ₽ | Всегда |
| Цена для резидентов/членов | `tariff.member_price` ₽ (зелёный) | Всегда |
| Плашка «Вступите» | CTA-блок экономии | Только гостям и пользователям без подписки |
| Кнопка | «Зарегистрироваться и оплатить» | Если `seats_available > 0` |

Если несколько тарифов — отображаются карточки рядом (2-3 в ряд на desktop, стек на mobile). Пользователь выбирает один.

Плашка «Вступите в ассоциацию» видна только гостям и пользователям без активной подписки. Ссылка «Подробнее о членстве» → `/register` (гость) или `/cabinet/payments` (авторизованный врач).

**Не показывать:** количество оставшихся мест (capacity, places_left), планируемую вместимость.

**Места:** если API возвращает `seats_available = 0` → текст «Места закончились», кнопка disabled. Если `seats_limit = null` → не отображать блок мест.

#### Блок 2a: Логика кнопки «Зарегистрироваться и оплатить»

> **ОБНОВЛЕНО** — Регистрация на мероприятие использует 3-сценарный flow с верификацией email. Подробная спецификация: `docs/UPDATED_EVENT_REGISTRATION_MODAL.md`.

**При нажатии на «льготную цену» (кнопка / чекбокс):**

| Doctor, активная подписка       | Цена переключается на member_price                              |

#### Модалка регистрации на мероприятие (state machine)

Модалка имеет 3 состояния. Переход определяется полем `action` в ответе `POST /register`.

```
[FORM] ──POST /register──→ action=null → [REDIRECT на payment_url]
                          → action="verify_existing" → [LOGIN]
                          → action="verify_new_email" → [CODE_INPUT]

[LOGIN] ──POST /auth/login──→ повторный POST /register с JWT → [REDIRECT]

[CODE_INPUT] ──POST /confirm-guest-registration──→ [REDIRECT на payment_url]
```

---

##### Состояние 1: FORM (начальная форма для неавторизованного)

| Поле            | Тип инпута            | Обязательно | Валидация             |
|-----------------|-----------------------|-------------|-----------------------|
| Email           | `<Input type="email">`| Да          | Формат email          |
| ФИО             | `<Input>`             | Нет         | Max 300               |
| Место работы    | `<Input>`             | Нет         | Max 255               |
| Специализация   | `<Input>`             | Нет         | Max 255               |
| Email для чека  | `<Input type="email">`| Нет         | Формат email (показывается по checkbox) |

Подпись: «После оплаты на указанный email будет отправлен временный пароль для доступа к личному кабинету».

**Request:**
```json
{
  "tariff_id": "UUID",
  "idempotency_key": "crypto.randomUUID()",
  "guest_email": "user@example.com",
  "guest_full_name": "Иван Иванов",
  "guest_workplace": "Клиника А",
  "guest_specialization": "Трихология",
  "fiscal_email": "user@example.com"
}
```

**Response (`RegisterForEventResponse`):**
```json
{
  "registration_id": "UUID | null",
  "payment_url": "string | null",
  "applied_price": "float | null",
  "is_member_price": "bool | null",
  "action": "null | 'verify_existing' | 'verify_new_email'",
  "masked_email": "string | null"
}
```

**Обработка ответа:**
- `action === null` + `payment_url` → `window.location.href = payment_url`
- `action === "verify_existing"` → переключить модалку в состояние LOGIN
- `action === "verify_new_email"` → переключить модалку в состояние CODE_INPUT

---

##### Состояние 2: LOGIN (email уже в базе — нужно войти)

Показывается когда `action === "verify_existing"`. Текст: «Этот email уже зарегистрирован. Войдите в аккаунт {masked_email}, чтобы продолжить.»

| Поле    | Тип инпута              | Обязательно |
|---------|-------------------------|-------------|
| Пароль  | `<Input type="password">` | Да        |

Ссылка «Забыли пароль?» → `/forgot-password`

**Логика:**
1. `POST /api/v1/auth/login { email, password }` → получить `access_token`
2. Повторить `POST /events/{id}/register` с JWT → получить `payment_url`
3. `window.location.href = payment_url`

**Ошибка:** 401 → toast «Неверный пароль»

---

##### Состояние 3: CODE_INPUT (новый email — ввод 6-значного кода)

Показывается когда `action === "verify_new_email"`. Текст: «Мы отправили код подтверждения на {masked_email}»

| Элемент | Описание |
|---------|----------|
| Код     | 6 отдельных `<input>` (auto-focus, paste support, auto-submit при заполнении всех 6) |
| Таймер  | Обратный отсчёт 10 минут. По истечении: «Код истёк» + кнопка «Отправить новый код» |
| Кнопка «Отправить повторно» | Disabled 60 секунд после отправки. Повторный `POST /register` с тем же email. Max 3 за 10 мин |

**Request (подтверждение):**
```json
POST /api/v1/events/{event_id}/confirm-guest-registration
{
  "email": "new@example.com",
  "code": "123456",
  "tariff_id": "UUID",
  "idempotency_key": "crypto.randomUUID()",
  "guest_full_name": "Иван Иванов",
  "guest_workplace": "...",
  "guest_specialization": "...",
  "fiscal_email": "..."
}
```

**При успехе (201):** `window.location.href = response.payment_url`

---

##### Ошибки регистрации (все состояния)

| Ошибка | HTTP | Действие фронта |
|--------|------|-----------------|
| Мест нет | 422 | Toast «Места закончились», закрыть модалку |
| Нет email (гость без JWT) | 422 | Подсветить поле email |
| Неверный код | 422 | Toast «Неверный код. Осталось N попыток», очистить поля |
| Код истёк | 422 | Toast «Код истёк», показать кнопку «Отправить новый код» |
| Лимит попыток (5) | 422 | Toast «Слишком много попыток, запросите новый код» |
| Лимит отправки кодов (3/10мин) | 422 | Toast «Попробуйте позже» |
| Тариф не найден | 404 | Toast «Тариф не найден» |
| Регистрация закрыта | 422 | Toast + disabled кнопку |

Для обоих вариантов фронтенд использует одно поле `video_playback_url` — разницу обрабатывает react-player автоматически.

#### 5.7.1 Рендеринг контентных блоков (Content Blocks)

> **NOT IMPLEMENTED** — API для Content Blocks не реализован на бекенде. Поле `content_blocks` пока не приходит в ответах публичного API (`/doctors/{id}`, `/events/{slug}`, `/articles/{slug}`). Модель `ContentBlock` существует в БД, но CRUD-эндпоинты не созданы. **Пропустить при разработке, вернуться после доработки бекенда.** Компонент рендеринга можно подготовить заранее по спецификации ниже.

Блоки приходят в поле `content_blocks` ответа API. Рендер по `block_type`:

| block_type | Рендер |
|------------|--------|
| `text` | Заголовок (h2/h3) + HTML-контент в `prose` |
| `image` | `<img>` с `media_url`, alt из `block_metadata.alt`, подпись из `block_metadata.caption` |
| `video` | Embed: YouTube/VK/Rutube по `media_url` или `block_metadata.provider` |
| `gallery` | Сетка изображений из `block_metadata.images[]` (url, alt) |
| `link` | Кнопка/ссылка: `link_label` → `link_url` |

**Фильтрация по device_type:** на mobile скрывать блоки с `device_type=desktop`, на desktop — с `device_type=mobile`. `both` — показывать везде.

**Локаль:** использовать блоки с `locale`, совпадающим с текущей локалью страницы (по умолчанию `ru`).

**Порядок:** блоки уже отсортированы по `sort_order` в ответе API.

- `GET /api/v1/payments/{id}/receipt` → открыть `receipt_url` в новой вкладке
- Если 404 → toast «Чек ещё формируется. Попробуйте позже»

**Пагинация:** серверная, limit = 20, offset-based.

**Empty state:** «У вас ещё нет платежей».
