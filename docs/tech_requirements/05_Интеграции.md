## 1. Платёжная система — ЮKassa (YooKassa)

> **Детальное описание вариантов интеграции, код и архитектура** — см. [payment_integration_variants.md](./payment_integration_variants.md).
> В данном разделе — общее описание и ключевые решения.

### 1.1 Описание

**Рекомендация:** использовать **ЮKassa** (ранее Яндекс.Касса) в качестве платёжного агрегатора. Деньги принимаются через ЮKassa, а вывод средств настраивается на расчётный счёт в ПСБ Банке.

> **Особенности НКО (Некоммерческая организация):**
> - Членские и вступительные взносы **не облагаются налогом** (пп. 1 п. 2 ст. 251 НК РФ)
> - Фискальные чеки для взносов **не требуются** (п. 2 ст. 1.2 54-ФЗ)
> - Оплата мероприятий (конференции, семинары) — это **реализация услуг**, чеки **обязательны**
> - Для подключения ЮKassa потребуется: устав НКО, выписка из ЕГРЮЛ, приказ о назначении руководителя

**Обоснование выбора ЮKassa вместо прямого эквайринга ПСБ:**

| Критерий                 | ЮKassa                                      | Прямой эквайринг ПСБ                    |
|--------------------------|----------------------------------------------|------------------------------------------|
| Документация             | Отличная, публичная, с примерами на Python   | Закрытая, по запросу через менеджера      |
| SDK                      | Официальный `yookassa-sdk-python`            | Нет SDK, только HTTP API                  |
| Фискализация (54-ФЗ)     | Встроенная через партнёров (АТОЛ, Оранж Дата)| Отдельная интеграция                      |
| Подключение              | Онлайн, 1-3 дня                              | Через банк, 1-3 недели                    |
| Способы оплаты           | Карты, SBP, ЮMoney, Apple/Google Pay          | Только карты                              |
| Комиссия                 | 2.8-3.5%                                     | ~2.0-2.5%                                |
| Вывод на ПСБ              | Настраивается через ЛК ЮKassa               | Напрямую                                  |
| Тестовый режим            | Полноценная песочница                         | Ограниченный тестовый стенд               |

**Метод интеграции:** REST API v3 + webhook. Не redirect — используем **встраиваемую форму (ЮKassa Widget)** на фронте, либо **redirect на hosted-страницу ЮKassa**.

### 1.2 Flow инициации платежа


### 1.3 Создание платежа — запрос к ЮKassa API

**Endpoint:** `POST https://api.yookassa.ru/v3/payments`

**Headers:**
```
Authorization: Basic {base64(shop_id:secret_key)}
Idempotence-Key: {uuid}
Content-Type: application/json
```

**Request body (подписка / взнос — БЕЗ чека):**

> Взносы НКО не фискализируются (251 НК РФ, п. 2 ст. 1.2 54-ФЗ) — объект `receipt` **не передаётся**.

### 1.4 Webhook обработка

**Endpoint на backend:** `POST /api/v1/webhooks/yookassa`

**ЮKassa вызывает webhook** при изменении статуса платежа. Формат тела запроса:

```json
{
  "type": "notification",
  "event": "payment.succeeded",
  "object": {
    "id": "2a47e7d2-000f-5000-a000-1d5e1c8e1a5f",
    "status": "succeeded",
    "amount": { "value": "10000.00", "currency": "RUB" },
    "payment_method": {
      "type": "bank_card",
      "id": "...",
      "saved": false,
      "card": { "last4": "4444", "expiry_month": "12", "expiry_year": "2028" }
    },
    "created_at": "2026-03-02T14:30:00.000+00:00",
    "captured_at": "2026-03-02T14:30:05.000+00:00",
    "metadata": {
      "internal_payment_id": "uuid",
      "product_type": "entry_fee",
      "user_id": "uuid"
    },
    "paid": true,
    "refundable": true,
    "receipt_registration": "succeeded"
  }
}
```

**Верификация подписи:**

ЮKassa уведомления подписываются. Способ верификации:

```python
import hmac
import hashlib

def verify_yookassa_webhook(body: bytes, signature: str, secret_key: str) -> bool:
    """
    ЮKassa присылает IP с фиксированного набора адресов.
    Основная защита — проверка IP + получение объекта заново через GET API.
    """
    pass

async def handle_webhook(notification: dict) -> None:
    payment_id = notification["object"]["id"]

    # Повторно получить платёж через API для верификации
    yookassa_payment = await yookassa_client.get_payment(payment_id)
    if yookassa_payment.status != notification["object"]["status"]:
        raise SecurityError("Payment status mismatch")
```

**Рекомендуемый подход:** IP whitelist + повторный GET-запрос к API ЮKassa для валидации данных (double-check pattern). IP-адреса ЮKassa: `185.71.76.0/27`, `185.71.77.0/27`, `77.75.153.0/25`, `77.75.156.11`, `77.75.156.35`.

**Обработка статусов:**

| event ЮKassa            | Действия на backend                                                |
|--------------------------|--------------------------------------------------------------------|
| `payment.succeeded`      | payment → succeeded, активация подписки/регистрации, чек **только для мероприятий**, уведомления |
| `payment.canceled`       | payment → failed, subscription → cancelled                        |
| `payment.waiting_for_capture` | Не используется (capture=true автоматически)                  |
| `refund.succeeded`       | payment → refunded / partially_refunded, чек возврата для мероприятий |

**Пошаговая логика при `payment.succeeded`:**

```
1. Найти payment по external_payment_id = object.id
2. ИДЕМПОТЕНТНОСТЬ: если payment.status уже 'succeeded' → return 200
3. Обернуть в транзакцию:
   a. payment.status → succeeded
   b. payment.paid_at = object.captured_at
   c. Если product_type IN (entry_fee, subscription):
      - subscription.status → active
      - subscription.starts_at = now() (или = prev.ends_at если продление)
      - subscription.ends_at = starts_at + plan.duration_months
      - doctor_profiles.is_public = true
      - Чек НЕ формируется (взносы НКО, 251 НК РФ)
   d. Если product_type = event:
      - event_registration.status → confirmed
      - Atomic SQL: UPDATE events SET seats_taken = seats_taken + 1 WHERE id = event_id
      - Сформировать фискальный чек (receipt_type = 'payment')
4. После коммита транзакции (side effects через TaskIQ):
   a. Уведомить пользователя (email + Telegram)
   b. Уведомить админов (Telegram-бот)
   c. Если есть telegram_binding → добавить в канал
```

### 1.5 Идемпотентность

**На стороне создания платежа:**
- Клиент генерирует `idempotency_key` (UUID v4) и передаёт в запросе
- Backend сохраняет ключ в Redis: `idempotency:pay:{user_id}:{key}` → TTL 24ч, значение = JSON response
- Если ключ уже есть → вернуть сохранённый ответ
- ЮKassa также поддерживает `Idempotence-Key` — передаём наш ключ

**На стороне webhook:**
- Проверка `payment.status` в БД перед обновлением
- Partial unique index `external_payment_id` предотвращает дубли
- Всегда возвращать 200, даже если уже обработано — иначе ЮKassa будет повторять

### 1.5.1 Redis-верификация для гостевой регистрации на мероприятие

> **Реализовано.** Верификация email гостя при регистрации на мероприятие использует Redis для временного хранения кодов и лимитов.

**Redis-ключи:**

| Ключ | Тип | TTL | Описание |
|------|------|-----|----------|
| `event_reg_verify:{email}` | String (JSON) | 600 сек | `{ "code": "123456", "tariff_id": "...", "event_id": "..." }` |
| `event_reg_attempts:{email}` | Integer | 600 сек | Счётчик неудачных попыток ввода кода. Max 5 |
| `event_reg_send_count:{email}` | Integer | 600 сек | Счётчик отправок кодов на email. Max 3 |

**Flow:**

```
1. POST /events/{id}/register { guest_email: "new@..." }
   → Backend: INCR event_reg_send_count:{email}
   → Если > 3 → 422 SEND_LIMIT_EXCEEDED
   → Генерация 6-значного цифрового кода (secrets.choice)
   → SET event_reg_verify:{email} '{"code":"...","tariff_id":"...","event_id":"..."}' EX 600
   → TaskIQ: send_event_verification_code(email, code, event_title)
   → Response: { action: "verify_new_email", masked_email: "n***@..." }

2. POST /events/{id}/confirm-guest-registration { email, code, ... }
   → GET event_reg_verify:{email} → парсинг JSON
   → INCR event_reg_attempts:{email}
   → Если > 5 → 422 TOO_MANY_ATTEMPTS
   → Если code не совпадает → 422 INVALID_CODE
   → Если совпадает:
     → Создать user (non_doctor, temp password)
     → Создать registration + payment → YooKassa → payment_url
     → DEL event_reg_verify:{email}, event_reg_attempts:{email}, event_reg_send_count:{email}
     → TaskIQ: send_guest_account_created(email, temp_password, event_title, frontend_url)
     → Response: { registration_id, payment_url, ... }
```

### 1.9 Обработка ошибок и edge cases

| Ситуация                                      | Решение                                                               |
|-----------------------------------------------|-----------------------------------------------------------------------|
| ЮKassa API недоступна при создании платежа    | Вернуть 503, клиент повторяет через 5 сек. Retry 3 раза с backoff    |
| Webhook не дошёл (ЮKassa retry до 24ч)        | Backend всегда отвечает 200. ЮKassa повторяет: через 5мин, 15мин, 1ч, 3ч, 6ч, 24ч |
| Пользователь закрыл страницу оплаты           | Платёж в статусе `pending`. Cron-задача: отменять pending-платежи старше 1 часа через API ЮKassa |
| Двойной webhook                               | Идемпотентность: проверка status перед обработкой                     |
| Несовпадение суммы                            | Сравнить amount из webhook с amount в payments. При расхождении — логировать, не обрабатывать |
| Оплата пришла для уже cancelled subscription  | Выполнить refund через API ЮKassa                                     |

## 2. Онлайн-касса / фискализация (54-ФЗ)

### 2.1 Описание

**Рекомендация:** использовать **встроенную фискализацию ЮKassa**. При создании платежа через API ЮKassa передаётся объект `receipt` — ЮKassa автоматически отправляет данные в подключённую ККТ (АТОЛ Онлайн, Оранж Дата или CloudKassir), и чек формируется без дополнительной интеграции.

**Обоснование:** отдельная интеграция с АТОЛ / Чекко / CloudKassir нужна только если платёжная система не поддерживает фискализацию. ЮKassa поддерживает — дублировать не нужно.

**Партнёр для ККТ:** **АТОЛ Онлайн** — крупнейший российский оператор, интегрирован с ЮKassa напрямую.

> **Ключевое правило фискализации для НКО:**
>
> | Тип платежа | Чек (54-ФЗ) | Основание |
> |---|---|---|
> | Вступительный взнос | **НЕ нужен** | пп. 1 п. 2 ст. 251 НК РФ, п. 2 ст. 1.2 54-ФЗ |
> | Ежегодный членский взнос | **НЕ нужен** | пп. 1 п. 2 ст. 251 НК РФ, п. 2 ст. 1.2 54-ФЗ |
> | Оплата мероприятия | **ОБЯЗАТЕЛЕН** | Реализация услуг, 54-ФЗ |
>
> Таким образом, объект `receipt` передаётся в ЮKassa **только при оплате мероприятий** (`product_type = 'event'`).

### 2.2 Flow формирования чека

```
Создание платежа в ЮKassa
  │  (receipt передаётся в теле запроса)
  ▼
ЮKassa принимает оплату
  │
  ▼
ЮKassa автоматически отправляет данные в АТОЛ
  │
  ▼
АТОЛ формирует фискальный чек и регистрирует в ФНС
  │
  ▼
АТОЛ возвращает фискальные данные в ЮKassa
  │
  ▼
ЮKassa отправляет чек покупателю на email
  │
  ▼
В webhook payment.succeeded:
  object.receipt_registration = "succeeded"
  │
  ▼
Backend получает webhook → сохраняет receipt данные
  │
  ▼
Дополнительно: GET /v3/receipts/{payment_id}
  → Получить полные фискальные реквизиты
  → Сохранить в таблицу receipts
```

### 2.3 Обязательные реквизиты чека

Передаются при создании платежа в объекте `receipt`:

| Реквизит            | Поле                        | Значение для нашего проекта                         |
|---------------------|-----------------------------|------------------------------------------------------|
| Email покупателя    | `customer.email`            | email пользователя                                   |
| Телефон (опц.)      | `customer.phone`            | Если доступен                                        |
| Наименование        | `items[].description`       | «Вступительный взнос...» / «Участие в конференции...» |
| Количество          | `items[].quantity`          | `"1.00"`                                              |
| Цена                | `items[].amount.value`      | Сумма платежа                                        |
| Ставка НДС          | `items[].vat_code`          | `1` (без НДС, если организация на УСН)               |
| Предмет расчёта     | `items[].payment_subject`   | `"service"`                                           |
| Способ расчёта      | `items[].payment_mode`      | `"full_payment"`                                      |

**Важно:** `vat_code` зависит от системы налогообложения организации. Для НКО на УСН обычно `1` (без НДС).

### 2.4 Получение фискальных данных

После успешной фискализации данные можно получить:

### 2.5 Сохранение чека в БД

> Чеки создаются **только для мероприятий** (`product_type = 'event'`). Для взносов чеки не формируются.

```
После получения webhook payment.succeeded (если product_type = 'event'):
  1. Создать запись в receipts (status='pending', receipt_type='payment', provider_receipt_id=null)
  2. Запустить TaskIQ-задачу fetch_receipt_data:
     a. GET /v3/receipts?payment_id={yookassa_id}
     b. Если receipt_registration = 'succeeded':
        - receipts.provider_receipt_id = receipt_id из ЮKassa
        - receipts.fiscal_number = fiscal_storage_number
        - receipts.fiscal_document = fiscal_document_number
        - receipts.fiscal_sign = fiscal_attribute
        - receipts.receipt_url = формируется из ОФД (nalog.ru)
        - receipts.receipt_data = полный JSON ответа
        - receipts.status = 'succeeded'
     c. Если receipt_registration = 'pending':
        - Повторить через 30 секунд (retry 5 раз)
     d. Если ошибка:
        - receipts.error_message = описание ошибки
        - receipts.status = 'failed'

При возврате (refund.succeeded для мероприятий):
  1. Создать receipts (receipt_type='refund', status='pending')
  2. Аналогичная TaskIQ-задача для получения фискальных данных чека возврата
```

**Формирование URL чека на сайте ФНС:**
```
https://receipt.taxcom.ru/v01/show?fn={fiscal_storage_number}&fp={fiscal_attribute}&i={fiscal_document_number}
```

Или через OFD:
```
https://ofd.ru/rec/{fiscal_storage_number}/{fiscal_document_number}/{fiscal_attribute}
```

### 2.6 Что делать если чек не сформировался

```
1. receipt_registration = 'pending' после 5 retry:
   → Логировать в Sentry / мониторинг
   → receipts.status = 'failed'
   → Уведомить администратора
   → Вручную пробить чек через ЛК ЮKassa / АТОЛ

2. receipt_registration = 'canceled':
   → receipts.status = 'failed'
   → Уведомить администратора
   → Требуется ручное вмешательство

3. Чек можно пробить повторно через API:
   POST /v3/receipts
   { "type": "payment", "payment_id": "...", ... }
```

```python
async def add_to_channel(tg_user_id: int) -> bool:
```

```python
# Вызывается при:
```

```python
    if not binding or not binding.tg_chat_id:
        return False

            parse_mode="HTML",
            reply_markup=keyboard,
```

### 6.4 Генерация подписанных URL

```python
# app/services/file_service.py

import aioboto3
from botocore.config import Config

class FileService:
    def __init__(self):
        self.session = aioboto3.Session()

    async def generate_presigned_url(
        self,
        key: str,
        expires_in: int = 600,
        content_type: str | None = None,
    ) -> str:
        async with self.session.client(
            "s3",
            endpoint_url=settings.S3_ENDPOINT_URL,
            aws_access_key_id=settings.S3_ACCESS_KEY,
            aws_secret_access_key=settings.S3_SECRET_KEY,
            region_name=settings.S3_REGION,
            config=Config(signature_version="s3v4"),
        ) as s3:
            params = {
                "Bucket": settings.S3_BUCKET,
                "Key": key,
            }
            if content_type:
                params["ResponseContentType"] = content_type

            url = await s3.generate_presigned_url(
                "get_object",
                Params=params,
                ExpiresIn=expires_in,
            )
            return url

    async def upload_file(
        self,
        key: str,
        file_data: bytes,
        content_type: str,
        acl: str = "private",
    ) -> str:
        async with self.session.client(
            "s3",
            endpoint_url=settings.S3_ENDPOINT_URL,
            aws_access_key_id=settings.S3_ACCESS_KEY,
            aws_secret_access_key=settings.S3_SECRET_KEY,
            region_name=settings.S3_REGION,
        ) as s3:
            await s3.put_object(
                Bucket=settings.S3_BUCKET,
                Key=key,
                Body=file_data,
                ContentType=content_type,
                ACL=acl,
            )
            if acl == "public-read":
                return f"{settings.S3_PUBLIC_URL}/{key}"
            return key

    async def delete_file(self, key: str) -> None:
        async with self.session.client("s3", ...) as s3:
            await s3.delete_object(Bucket=settings.S3_BUCKET, Key=key)
```

```python
MAGIC_BYTES = {
    "image/jpeg": [b"\xff\xd8\xff"],
    "image/png": [b"\x89PNG\r\n\x1a\n"],
    "application/pdf": [b"%PDF"],
    "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet": [b"PK\x03\x04"],
    "video/mp4": [b"\x00\x00\x00\x18ftypmp4", b"\x00\x00\x00\x1cftypisom", b"\x00\x00\x00\x20ftypisom"],
    "video/webm": [b"\x1a\x45\xdf\xa3"],
    "video/quicktime": [b"\x00\x00\x00\x14ftypqt", b"\x00\x00\x00\x08wide"],
}

def validate_file_type(file_data: bytes, expected_types: list[str]) -> str | None:
    for mime_type in expected_types:
        for magic in MAGIC_BYTES.get(mime_type, []):
            if file_data[:len(magic)] == magic:
                return mime_type
    return None
```

### 6.6 Загрузка видеозаписей

Видеофайлы могут быть до 2 ГБ — стандартный `put_object` не подходит. Используется **S3 multipart upload**.

**Стратегия загрузки:**

```python
async def upload_video(
    key: str,
    file: UploadFile,
    content_type: str,
) -> dict:
    """
    Для файлов > 100 MB — S3 multipart upload (чанки по 10 MB).
    Для файлов <= 100 MB — обычный put_object.
    """
    CHUNK_SIZE = 10 * 1024 * 1024  # 10 MB

    async with s3_session.client("s3", ...) as s3:
        if file.size > 100 * 1024 * 1024:
            mpu = await s3.create_multipart_upload(
                Bucket=settings.S3_BUCKET,
                Key=key,
                ContentType=content_type,
            )
            parts = []
            part_number = 1
            while chunk := await file.read(CHUNK_SIZE):
                part = await s3.upload_part(
                    Bucket=settings.S3_BUCKET,
                    Key=key,
                    UploadId=mpu["UploadId"],
                    PartNumber=part_number,
                    Body=chunk,
                )
                parts.append({"PartNumber": part_number, "ETag": part["ETag"]})
                part_number += 1

            await s3.complete_multipart_upload(
                Bucket=settings.S3_BUCKET,
                Key=key,
                UploadId=mpu["UploadId"],
                MultipartUpload={"Parts": parts},
            )
        else:
            data = await file.read()
            await s3.put_object(
                Bucket=settings.S3_BUCKET,
                Key=key,
                Body=data,
                ContentType=content_type,
            )
    return {"key": key, "size": file.size, "content_type": content_type}
```

**Поддерживаемые форматы:** MP4 (рекомендуемый), WebM, MOV.

**Ключ в S3:** `events/{event_id}/recordings/{uuid}.{ext}`

**При удалении записи (video_source = uploaded):** удалить файл из S3 через `delete_object`.

### 6.7 Обработка изображений (Pillow)

```python
from PIL import Image
import io

async def process_photo(
    file_data: bytes,
    max_size: tuple[int, int] = (800, 800),
    thumb_size: tuple[int, int] = (200, 200),
    quality: int = 85,
) -> tuple[bytes, bytes]:
    img = Image.open(io.BytesIO(file_data))
    img = img.convert("RGB")

    img.thumbnail(max_size, Image.Resampling.LANCZOS)
    main_buffer = io.BytesIO()
    img.save(main_buffer, format="JPEG", quality=quality, optimize=True)

    thumb = img.copy()
    thumb.thumbnail(thumb_size, Image.Resampling.LANCZOS)
    thumb_buffer = io.BytesIO()
    thumb.save(thumb_buffer, format="JPEG", quality=quality, optimize=True)

    return main_buffer.getvalue(), thumb_buffer.getvalue()
```
