# Техническое задание: проектирование реляционной базы данных

## Проект: «Ассоциация трихологов»

**СУБД:** PostgreSQL 16+  
**Нормализация:** 3NF  
**Первичные ключи:** UUID v7 (сортируемые по времени)  
**Временные метки:** TIMESTAMPTZ (UTC)  
**Денежные поля:** NUMERIC(12,2)  

---

## 1. Справочники и аутентификация

---

### Таблица: roles

Назначение: справочник ролей пользователей в системе.

| Поле       | Тип          | PK | FK | NOT NULL | UNIQUE | Default | Описание                               |
|------------|--------------|----|----|----------|--------|---------|----------------------------------------|
| id         | SMALLINT     | ✓  |    | ✓        | ✓      | GEN     | Идентификатор роли                     |
| name       | VARCHAR(50)  |    |    | ✓        | ✓      |         | Системное имя (doctor, non_doctor, admin, accountant, manager) |
| title      | VARCHAR(100) |    |    | ✓        |        |         | Отображаемое название роли             |
| created_at | TIMESTAMPTZ  |    |    | ✓        |        | now()   | Дата создания записи                   |

Связи: один-ко-многим → `user_roles`  
Индексы: PK (`id`), UNIQUE (`name`)

---

### Таблица: cities

Назначение: справочник городов, управляемый администратором.

| Поле       | Тип          | PK | FK | NOT NULL | UNIQUE | Default | Описание                   |
|------------|--------------|----|----|----------|--------|---------|----------------------------|
| id         | UUID         | ✓  |    | ✓        | ✓      | gen_random_uuid() | Идентификатор города |
| name       | VARCHAR(255) |    |    | ✓        | ✓      |         | Название города            |
| sort_order | INT          |    |    | ✓        |        | 0       | Порядок в списке           |

| created_at | TIMESTAMPTZ  |    |    | ✓        |        | now()   | Дата создания              |

Связи: один-ко-многим → `doctor_profiles`  
Индексы: PK (`id`), UNIQUE (`name`), INDEX (`is_active, sort_order`)

---

### Таблица: moderation_statuses

Назначение: справочник статусов модерации.

| Поле  | Тип          | PK | FK | NOT NULL | UNIQUE | Default | Описание                                        |
|-------|--------------|----|----|----------|--------|---------|--------------------------------------------------|
| id    | SMALLINT     | ✓  |    | ✓        | ✓      | GEN     | Идентификатор                                    |

## 2. Пользователи и роли

---

### Таблица: users

Назначение: базовая таблица всех аккаунтов в системе — врачи, не-врачи, администраторы, бухгалтеры, менеджеры.

| Поле              | Тип          | PK | FK | NOT NULL | UNIQUE | Default | Описание                                   |
|-------------------|--------------|----|----|----------|--------|---------|--------------------------------------------|
| id                | UUID         | ✓  |    | ✓        | ✓      | gen_random_uuid() | Идентификатор пользователя    |
| email             | VARCHAR(255) |    |    | ✓        | ✓      |         | Email (логин)                              |
| password_hash     | VARCHAR(255) |    |    | ✓        |        |         | Хеш пароля (bcrypt/argon2)                 |
| email_verified_at | TIMESTAMPTZ  |    |    |          |        | NULL    | Дата подтверждения email                   |
| is_active         | BOOLEAN      |    |    | ✓        |        | true    | Аккаунт активен                            |
| last_login_at     | TIMESTAMPTZ  |    |    |          |        | NULL    | Время последнего входа                     |
| created_at        | TIMESTAMPTZ  |    |    | ✓        |        | now()   | Дата регистрации                           |
| updated_at        | TIMESTAMPTZ  |    |    | ✓        |        | now()   | Дата последнего обновления                 |

Связи:  
- один-ко-многим → `user_roles`  
- один-к-одному → `doctor_profiles`  
- один-ко-многим → `subscriptions`, `payments`, `event_registrations`, `notifications`, `votes`  
- один-к-одному → `telegram_bindings`  

Индексы: PK (`id`), UNIQUE (`email`), INDEX (`is_active`), INDEX (`created_at`)

---

### Таблица: user_roles

Назначение: связь пользователей с ролями (многие-ко-многим).

| Поле        | Тип         | PK | FK | NOT NULL | UNIQUE | Default | Описание                     |
|-------------|-------------|----|----|----------|--------|---------|------------------------------|
| user_id     | UUID        | ✓  | ✓  | ✓        |        |         | FK → users.id                |
| role_id     | SMALLINT    | ✓  | ✓  | ✓        |        |         | FK → roles.id                |
| assigned_at | TIMESTAMPTZ |    |    | ✓        |        | now()   | Дата назначения роли         |

PK: составной (`user_id`, `role_id`)  
FK:  
- `user_id` → `users.id` ON DELETE CASCADE ON UPDATE CASCADE  
- `role_id` → `roles.id` ON DELETE RESTRICT ON UPDATE CASCADE  

Связи: реализует M:N между `users` и `roles`  
Индексы: PK (`user_id, role_id`), INDEX (`role_id`)

## 13. Бизнес-правила на уровне БД

### CHECK-ограничения

```sql
-- payments: сумма всегда положительная
ALTER TABLE payments ADD CONSTRAINT chk_payments_amount_positive CHECK (amount > 0);

-- receipts: сумма всегда положительная
ALTER TABLE receipts ADD CONSTRAINT chk_receipts_amount_positive CHECK (amount > 0);

-- event_tariffs: цены неотрицательные
ALTER TABLE event_tariffs ADD CONSTRAINT chk_event_tariffs_price CHECK (price >= 0);
ALTER TABLE event_tariffs ADD CONSTRAINT chk_event_tariffs_member_price CHECK (member_price >= 0);
ALTER TABLE event_tariffs ADD CONSTRAINT chk_event_tariffs_seats CHECK (seats_limit IS NULL OR seats_limit > 0);
ALTER TABLE event_tariffs ADD CONSTRAINT chk_event_tariffs_seats_taken CHECK (seats_taken >= 0);

-- subscriptions: дата окончания после даты начала
ALTER TABLE subscriptions ADD CONSTRAINT chk_subscriptions_dates
  CHECK (starts_at IS NULL OR ends_at IS NULL OR ends_at > starts_at);

```

### Триггеры

```sql
-- 1. Автообновление updated_at
-- Триггерная функция для автоматического обновления поля updated_at
-- Применяется к таблицам: users, doctor_profiles, subscriptions, payments, articles,
-- organization_documents, pages_seo, events, plans

CREATE OR REPLACE FUNCTION trigger_set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Пример применения (аналогично для всех таблиц с updated_at):
CREATE TRIGGER trg_users_updated_at
  BEFORE UPDATE ON users
  FOR EACH ROW EXECUTE FUNCTION trigger_set_updated_at();


-- 2. Контроль seats_taken при регистрации
-- При INSERT в event_registrations со статусом confirmed — инкремент seats_taken.
-- При UPDATE статуса на cancelled — декремент. Если seats_taken >= seats_limit — RAISE.

CREATE OR REPLACE FUNCTION trigger_event_registration_seats()
RETURNS TRIGGER AS $$
DECLARE
  v_limit INT;
  v_taken INT;
BEGIN
  IF TG_OP = 'INSERT' AND NEW.status = 'confirmed' THEN
    SELECT seats_limit, seats_taken INTO v_limit, v_taken
      FROM event_tariffs WHERE id = NEW.event_tariff_id FOR UPDATE;
    IF v_limit IS NOT NULL AND v_taken >= v_limit THEN
      RAISE EXCEPTION 'No seats available for this tariff';
    END IF;
    UPDATE event_tariffs SET seats_taken = seats_taken + 1 WHERE id = NEW.event_tariff_id;
  END IF;

  IF TG_OP = 'UPDATE'
     AND OLD.status = 'confirmed' AND NEW.status = 'cancelled' THEN
    UPDATE event_tariffs SET seats_taken = GREATEST(seats_taken - 1, 0)
      WHERE id = OLD.event_tariff_id;
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_event_registration_seats
  AFTER INSERT OR UPDATE ON event_registrations
  FOR EACH ROW EXECUTE FUNCTION trigger_event_registration_seats();


-- 3. Деактивация публичного профиля при истечении подписки
-- При UPDATE subscriptions.status → 'expired' → doctor_profiles.is_public = false

CREATE OR REPLACE FUNCTION trigger_subscription_expired()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.status = 'expired' AND OLD.status != 'expired' THEN
    UPDATE doctor_profiles SET is_public = false WHERE user_id = NEW.user_id;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_subscription_expired
  AFTER UPDATE ON subscriptions
  FOR EACH ROW EXECUTE FUNCTION trigger_subscription_expired();


-- 4. Обеспечение одного голоса за сессию (дублирует UNIQUE constraint как защита)
-- UNIQUE (voting_session_id, user_id) на таблице votes уже обеспечивает это.
-- Дополнительно можно добавить триггер для проверки что сессия в статусе 'active'.

CREATE OR REPLACE FUNCTION trigger_vote_session_active()
RETURNS TRIGGER AS $$
DECLARE
  v_status VARCHAR(20);
BEGIN
  SELECT status INTO v_status FROM voting_sessions WHERE id = NEW.voting_session_id;
  IF v_status != 'active' THEN
    RAISE EXCEPTION 'Voting session is not active';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_vote_session_active
  BEFORE INSERT ON votes
  FOR EACH ROW EXECUTE FUNCTION trigger_vote_session_active();
```

### Важные бизнес-правила (реализуются на уровне приложения с поддержкой БД)

| # | Правило | Реализация |
|---|---------|------------|
| 1 | Вступительный взнос оплачивается однократно и включает первый год подписки | CHECK в приложении: не более одного payment с типом `entry_fee` на пользователя. Подписка создаётся сразу на `duration_months` из тарифа |

| 7 | Сертификат генерируется при переводе `moderation_status_id` в «approved» | Триггер или event listener в приложении создаёт запись в `certificates` |
