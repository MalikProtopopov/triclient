## 1. Инфраструктура и общие паттерны

### 1.1 Middleware (критично)

**ТЗ (раздел 5.1):** Middleware должен проверять `access_token` и редиректить неавторизованных на `/admin/login`.

**Текущее состояние:** Middleware только редиректит `/admin` → `/admin/dashboard`. Проверки токена нет — авторизация делается в layout через `useAuth` (клиент).

**Доработка:**
- Добавить проверку JWT/cookie в middleware для защиты маршрутов
- Редирект неавторизованных на `/admin/login`
- Редирект авторизованных с `/admin/login` на `/admin/dashboard`
- Для роли Accountant: редирект с `/admin/dashboard` на `/admin/payments`

### 1.2 Skeleton loaders

**ТЗ (раздел 1.3):** Loading — skeleton-loader, повторяющий структуру контента, не спиннер на весь экран.

**Текущее состояние:** Используется `PageLoader` (спиннер) и в `DataTable` — 5 скелетон-полос.

**Доработка:**
- Заменить `PageLoader` на страничные skeleton-компоненты (карточки, таблицы)
- Добавить `DashboardSkeleton`, `TableSkeleton` и т.п.

### 1.3 Unsaved changes guard

**ТЗ (раздел 4.2):** При уходе со страницы с несохранёнными изменениями — confirm «У вас есть несохранённые изменения. Покинуть страницу?».

**Текущее состояние:** Не реализовано.

**Доработка:**
- Хук `useUnsavedChangesGuard(isDirty)` или `useBlocker` (React Router)
- Подключить на формах: мероприятия, статьи, документы, настройки

### 1.4 Keyboard shortcuts

**ТЗ (раздел 4.4):**

| Комбинация | Действие |
|------------|----------|
| Ctrl+S | Сохранить (на страницах форм) |
| Escape | Закрыть модалку |
| / | Фокус на поиск (на страницах списков) |

**Текущее состояние:** Не реализовано (Escape частично через Radix).

**Доработка:**
- Глобальный обработчик `useEffect` для Ctrl+S на формах
- Обработчик `/` для фокуса на поле поиска (врачи, платежи, уведомления)

### 1.5 Responsive

**ТЗ (раздел 4.5):** Минимальная ширина 1024px, сайдбар в hamburger при < 1280px.

**Текущее состояние:** Sidebar скрыт при `lg` (1024px), Header с hamburger показывается при `lg:hidden`. Breakpoint 1280px не проверен.

**Доработка:**
- Уточнить breakpoint сайдбара: `xl` (1280px) вместо `lg` (1024px)

## 2. Авторизация

### 2.1 Валидация reset-password

**ТЗ (раздел 3.1):** Пароль: min 8, max 128, 1 цифра + 1 буква.

**Текущее состояние:** Только `min(8)` в Zod.

**Доработка:**
```ts
password: z.string().min(8).max(128).regex(/^(?=.*[0-9])(?=.*[a-zA-Z]).+$/, "Минимум 1 цифра и 1 буква")
```

### 2.2 Обработка 403 ACCOUNT_DEACTIVATED

**ТЗ:** При 403 (ACCOUNT_DEACTIVATED) — toast «Аккаунт деактивирован».

**Текущее состояние:** Общий обработчик ошибок в api.ts, без отдельной обработки 403.

**Доработка:** В interceptor проверять `error.response?.data?.error?.code === 'ACCOUNT_DEACTIVATED'` и показывать соответствующий toast.

### 3.3 Вкладка «Контентные блоки» в карточке врача

**ТЗ (раздел 3.3, вкладка «Контентные блоки»):** `entity_type=doctor_profile`, `entity_id` = id профиля.

**Текущее состояние:** Вкладка «Контентные блоки» отсутствует в табах карточки врача.

**Текущее состояние:** В `ConfirmDialog` для одобрения этого текста нет.

**Доработка:** Добавить в description модалки.

## 4. Мероприятия

### 4.1 Фильтр «Период»

**ТЗ:** Фильтр периода: Предстоящие / Прошедшие / Все.

**Текущее состояние:** Только фильтр по статусу.

**Доработка:** Добавить `<Select>` с опциями периода и передавать в API.

### 4.2 Удаление мероприятия — проверка регистраций

**ТЗ:** Если есть confirmed-регистрации → toast «Сначала отмените регистрации».

**Текущее состояние:** Confirm dialog без проверки.

**Доработка:** Перед удалением проверять `registrations_count` (confirmed), при наличии показывать toast и не вызывать DELETE.

### 4.3 Секция «Фотогалереи» (только в edit mode)

**ТЗ (раздел 3.4, секция 3):** Список галерей, создание галереи, загрузка фото (multiple, max 50).

**Текущее состояние:** Не реализовано.

**Доработка:**
- Секция «Фотогалереи» в EventForm (только при редактировании)
- Модалка создания галереи (название, уровень доступа)
- Загрузка фото через FileUpload (multiple)

### 4.4 Секция «Контентные блоки» (только в edit mode)

**ТЗ (раздел 3.4, секция 4):** `entity_type=event`, `entity_id` = id мероприятия.

**Текущее состояние:** Не реализовано в EventForm.

**Доработка:**
- Добавить `ContentBlocksEditor` в форму мероприятия (только при edit)

### 4.5 Секция «Записи конференции»

**ТЗ (раздел 3.4, секция 5):** Таблица записей (название, источник, длительность, размер, доступ, статус). Модалка добавления: uploaded / external, URL или файл, уровень доступа.

**Текущее состояние:** Не реализовано.

**Доработка:**
- Секция «Записи конференции» в EventForm
- Таблица записей + модалка добавления/редактирования
- RadioGroup: uploaded / external
- FileUpload для видео (MP4/WebM/MOV, max 2 GB) или Input для URL

### 4.6 Кнопки «Сохранить черновик» и «Опубликовать»

**ТЗ:** Две отдельные кнопки вместо одной «Сохранить».

**Текущее состояние:** Одна кнопка «Сохранить» / «Создать».

**Доработка:** Две кнопки с разным `status` при отправке.

### 4.7 Тарифы — модалка вместо inline

**ТЗ:** Таблица тарифов + модалка добавления/редактирования. Несколько тарифов с разными названиями; у каждого — обычная цена и цена для резидентов/членов ассоциации. Поля: название, описание, условия, доп. детали, преимущества (BenefitsList).

**Текущее состояние:** Inline-редактирование тарифов в форме.

**Доработка:**
- Вынести тарифы в модалку
- Поля: название, описание, условия (кто может выбрать), доп. детали, обычная цена, цена для резидентов, преимущества, лимит мест
- BenefitsList: массив Input + кнопка «Добавить пункт»
- Drag-and-drop для порядка преимуществ (опционально)

## 6. Контент

### 6.1 Фильтр статей по статусу

**ТЗ:** Фильтр: Все / Черновик / Опубликовано / Архив.

**Текущее состояние:** Нет фильтра.

**Доработка:** Добавить `<Select>` над таблицей.

### 6.2 Документы организации — структура страниц

**ТЗ:** Отдельные страницы `/documents/new` и `/documents/[id]/edit`.

**Текущее состояние:** Одна страница со списком и модалками создания/редактирования.

**Доработка (опционально):** Можно оставить модалки, если UX устраивает. Либо перейти на отдельные страницы по ТЗ.

### 6.3 Документы — drag-and-drop порядка

**ТЗ:** Порядок можно менять drag-and-drop.

**Текущее состояние:** Список с кнопками вверх/вниз, без drag-and-drop.

**Доработка:** Подключить @dnd-kit или react-beautiful-dnd для перетаскивания.

### 6.4 Документы — выбор: WYSIWYG или PDF

**ТЗ:** Должно быть заполнено хотя бы одно: содержание или файл.

**Текущее состояние:** Только WYSIWYG в модалке, FileUpload для PDF не добавлен.

**Доработка:**
- Добавить FileUpload для PDF (max 20 MB)
- Валидация: content ИЛИ file_url обязательно

### 6.5 Контентные блоки — полная реализация

**ТЗ (раздел 3.6.1):**
- Типы: text, image, video, gallery, link
- Модалка редактирования для каждого типа
- Поля: Alt-текст, подпись (image); device_type
- API reorder для drag-and-drop

**Текущее состояние:** ContentBlocksEditor есть, но упрощённый (inline-редактирование, без модалок, без gallery, без device_type).

**Доработка:**
- Модалка редактирования блока вместо inline
- Тип gallery: загрузка нескольких изображений
- Поле device_type в модалке
- API `POST .../reorder` при изменении порядка

## 8. Голосование

### 8.1 Детальная страница `/admin/voting/[id]`

**ТЗ:** Отдельная страница с шапкой, блоком действий, результатами.

**Текущее состояние:** Результаты показываются в модалке из списка. Отдельной страницы нет.

**Доработка:**
- Создать `/admin/voting/[id]/page.tsx`
- Перенести логику результатов и действий на эту страницу
- Ссылка «Посмотреть» в списке ведёт на `/admin/voting/[id]`

### 8.2 Polling для активного голосования

**ТЗ:** При status = active результаты обновляются каждые 30 сек.

**Текущее состояние:** Нет polling.

**Доработка:** `refetchInterval: 30_000` в useQuery для страницы активного голосования.

### 8.3 Создание — описание кандидата

**ТЗ:** Для каждого кандидата — опциональное поле «Описание / программа».

**Текущее состояние:** Только выбор кандидатов чекбоксами, без описания.

**Доработка:** При выборе кандидата — опциональное Textarea для описания.
