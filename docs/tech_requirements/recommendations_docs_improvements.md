# Рекомендации по доработке технической документации
## Проект: «Ассоциация трихологов»

**Версия:** 1.0 | **Дата:** 2026-03-03 | **Статус:** Актуально

---

## Контекст: что изменилось после уточнений с клиентом

Перед разбором конкретных правок — список фактов, которые были выяснены и влияют на документацию:

1. У клиента уже есть расчётный счёт в ПСБ
2. У клиента уже зарегистрирована офлайн-ККТ (ПСБ Касса) с договором ОФД
3. Организация — МОО (межрегиональная общественная организация), зарегистрирована 01.12.2023
4. Предположительно УСН 6%, но факт подачи уведомления в ФНС нужно проверить
5. Членские и вступительные взносы НКО **не попадают под 54-ФЗ** — чеки не нужны
6. Оплата мероприятий физлицами — предпринимательская деятельность, чеки обязательны
7. Возраст организации (1,2 года) создаёт риск затянутой проверки при подключении ЮKassa
8. Дедлайн запуска — май 2026

## 1. Доработки ТЗ на базу данных (DB Schema v2.0)

### 1.1 Критические правки

#### 1.1.1 Поле `payments.product_type` — уточнить enum

**Текущее состояние:** enum `product_type` содержит значения `entry_fee | subscription | event`.

**Проблема:** значение `entry_fee` семантически включает и вступительный взнос, и первый год подписки. Это создаёт путаницу в логике фискализации — для вступительного взноса чек **не нужен**, тогда как нужно чётко понимать, когда создавать `receipts`.

**Рекомендация:** добавить бизнес-правило на уровне документации:

```
payments.product_type = 'entry_fee'     → чек НЕ формируется (receipts не создаётся)
payments.product_type = 'subscription'  → чек НЕ формируется (receipts не создаётся)
payments.product_type = 'event'         → чек ОБЯЗАТЕЛЕН (receipts создаётся)
```

Добавить в таблицу `payments` колонку-флаг для явности:

```sql
ALTER TABLE payments ADD COLUMN requires_receipt BOOLEAN NOT NULL DEFAULT false;
-- Заполняется автоматически при создании payment на уровне приложения:
-- product_type = 'event' → requires_receipt = true
-- product_type IN ('entry_fee', 'subscription') → requires_receipt = false
```

Это избавляет от необходимости «помнить» бизнес-правило в каждом месте кода.

---

#### 1.1.2 Таблица `receipts` — добавить поле `receipt_type`

**Проблема:** при возврате оплаты за мероприятие формируется чек возврата. Текущая схема не разделяет чек оплаты и чек возврата.

**Рекомендация:**

```sql
CREATE TYPE receipt_type AS ENUM (
  'payment',  -- чек прихода
  'refund'    -- чек возврата
);

ALTER TABLE receipts ADD COLUMN receipt_type receipt_type NOT NULL DEFAULT 'payment';
```

---

#### 1.1.3 Таблица `payments` — добавить поле `payment_provider`

Текущая документация написана под ПСБ, но после уточнений реальный выбор — ЮKassa или ПСБ. Схема должна быть нейтральной:

```sql
CREATE TYPE payment_provider AS ENUM (
  'yookassa',
  'psb',
  'manual'  -- ручной платёж, внесённый администратором
);

ALTER TABLE payments ADD COLUMN payment_provider payment_provider NOT NULL DEFAULT 'yookassa';
```

Это также позволит в будущем вести историю по нескольким провайдерам.

---

#### 1.1.4 Таблица `receipts` — добавить поле `provider_receipt_id`

У ЮKassa фискальный чек имеет собственный ID внутри системы (`rt-...`). У ПСБ при интеграции с внешней ККТ (АТОЛ) будет аналогичный идентификатор.

```sql
ALTER TABLE receipts ADD COLUMN provider_receipt_id VARCHAR(255) NULL;
-- Для ЮKassa: ID чека из GET /v3/receipts
-- Для АТОЛ: ID документа из API АТОЛ
```

---

### 1.2 Рекомендуемые правки

#### 1.2.1 Таблица `subscriptions` — добавить флаг `is_first_year`

При вступительном взносе первый год включён в стоимость. Важно отслеживать это для аналитики и для логики «нельзя оплатить annual_fee без entry_fee»:

```sql
ALTER TABLE subscriptions ADD COLUMN is_first_year BOOLEAN NOT NULL DEFAULT false;
-- true для подписки, созданной из entry_fee
-- false для всех последующих annual_fee
```

---

#### 1.2.2 Таблица `event_registrations` — добавить поле `fiscal_email`

Email для чека может отличаться от email аккаунта (особенно для гостей). Явно хранить его при создании регистрации:

```sql
ALTER TABLE event_registrations ADD COLUMN fiscal_email VARCHAR(255) NULL;
-- Заполняется при создании регистрации:
-- Для зарегистрированных: users.email
-- Для гостей: guest_email
```

---

#### 1.2.3 Открытый вопрос из документа — ответ по нескольким специализациям

В v2.0 зафиксирован открытый вопрос: «Нужна ли поддержка нескольких специализаций?». На данный момент достаточно одной FK-связи. **Рекомендация:** оставить одно поле `specialization_id` для MVP, но предусмотреть в архитектуре место для миграции (не делать hard-coded логику вокруг «одна специализация»).

---

#### 1.2.4 Таблица `doctor_profile_changes` — явно хранить список изменённых полей

Текущая структура хранит JSONB `changes` с новыми значениями. Администратору при модерации нужно видеть **что именно изменилось** (diff). Рекомендуется добавить:

```sql
ALTER TABLE doctor_profile_changes ADD COLUMN changed_fields TEXT[] NOT NULL DEFAULT '{}';
-- Пример: ['bio', 'city_id', 'clinic_name']
-- Заполняется на уровне приложения при создании черновика
```

## 2. Доработки ТЗ на API

### 2.1 Критические правки

#### 2.1.1 Endpoint `POST /api/v1/subscriptions/pay` — убрать упоминание ПСБ

В текущем ТЗ на API response явно ссылается на `payment.psbbank.ru`. Это нужно привести к нейтральному виду:

```json
// Было (привязка к ПСБ):
"payment_url": "https://payment.psbbank.ru/pay?order=..."

// Должно быть (нейтрально):
"payment_url": "https://yoomoney.ru/checkout/payments/v2/contract?orderId=..."
// или просто "payment_url": "string"
```

---

#### 2.1.2 Endpoint `POST /api/v1/webhooks/payment` — переименовать и разделить

Текущее ТЗ описывает один вебхук-эндпоинт под ПСБ. При любом провайдере должен быть свой путь:

```
// Вариант ЮKassa:
POST /api/v1/webhooks/yookassa

// Вариант ПСБ:
POST /api/v1/webhooks/psb
```

Разные провайдеры присылают вебхуки в разном формате, с разной схемой верификации. Один общий эндпоинт — архитектурная ошибка.

---

#### 2.1.3 Добавить endpoint для ручного добавления платежа администратором

Сценарий: клиент оплатил банковским переводом, менеджер хочет отметить оплату вручную. Этот сценарий не описан в текущем ТЗ.

```
POST /api/v1/admin/payments/manual
Access: Admin, Accountant

Request body:
{
  "user_id": "uuid",
  "amount": "number",
  "product_type": "entry_fee | subscription | event",
  "subscription_id": "uuid (если subscription)",
  "event_registration_id": "uuid (если event)",
  "description": "string",
  "paid_at": "datetime"
}
```

---

#### 2.1.4 Добавить endpoint `GET /api/v1/payments/{payment_id}/status`

Пользователь вернулся с платёжной страницы, фронтенд должен проверить актуальный статус. Это polling-endpoint:

```
GET /api/v1/payments/{payment_id}/status
Access: Авторизованный (свой платёж) или Admin

Response 200:
{
  "payment_id": "uuid",
  "status": "pending | processing | succeeded | failed | refunded",
  "amount": 10000.00,
  "paid_at": "2026-03-02T14:30:00Z | null"
}
```

Без этого эндпоинта фронтенд не сможет корректно отобразить результат оплаты — нельзя полагаться только на вебхуки.

---

### 2.2 Рекомендуемые правки

#### 2.2.1 Endpoint `POST /api/v1/events/{event_id}/register` — уточнить создание гостевого аккаунта

Текущее ТЗ: «Если guest → проверить email → если уже есть user, вернуть ошибку AUTH_REQUIRED».

Это неудобно для пользователя. **Рекомендация:** если email уже существует, предложить войти и завершить регистрацию авторизованным. Изменить код ошибки и текст:

```json
// Было:
"code": "AUTH_REQUIRED"

// Должно быть:
"code": "EMAIL_EXISTS_PLEASE_LOGIN",
"message": "Аккаунт с этим email уже существует. Войдите, чтобы зарегистрироваться на мероприятие"
```

---

#### 2.2.2 Добавить endpoint `POST /api/v1/payments/{payment_id}/refund`

Возврат оплаты за мероприятие описан в ТЗ на интеграции, но отсутствует как API-эндпоинт.

```
POST /api/v1/admin/payments/{payment_id}/refund
Access: Admin, Accountant

Request body:
{
  "amount": "number (optional, если частичный возврат)",
  "reason": "string (required)"
}

Response 200:
{
  "refund_id": "string",
  "status": "pending | succeeded",
  "amount": 5000.00
}
```

---

#### 2.2.3 Уточнить логику `GET /api/v1/onboarding/status` — добавить шаг `await_payment`

Текущий список `next_step` не включает состояние «одобрен, ожидает оплаты». Цепочка должна быть:

```
verify_email → choose_role → fill_profile → upload_documents
→ submit → await_moderation → await_entry_fee_payment → completed
```

Добавить значение `await_entry_fee_payment` для случая когда moderation_status = `approved` и `has_paid_entry_fee = false`.

## 3. Доработки ТЗ на интеграции

### 3.1 Критические правки

#### 3.1.1 Весь раздел «Платёжная система» — переработан отдельным документом

Текущее ТЗ на интеграции написано только под ЮKassa с кратким упоминанием ПСБ. С учётом того, что у клиента есть счёт в ПСБ и уже зарегистрированная ККТ — оба варианта равноправны и заслуживают полного описания. **Это вынесено в отдельный документ** `payment_integration_variants.md`.

---

#### 3.1.2 Раздел «Фискализация» — полностью пересмотреть

Текущее ТЗ описывает «Чеки от ЮKassa» как единственный вариант. Теперь известно:

- У клиента **уже есть ККТ** (зарегистрирована, есть договор с ОФД)
- При выборе ПСБ-эквайринга — использовать существующую ККТ через API
- При выборе ЮKassa — можно использовать либо «Чеки от ЮKassa», либо подключить существующую ККТ как внешний сервис

Таблица вариантов фискализации с учётом новых данных:

| Вариант | Стоимость | Дополнительная интеграция | Использует существующую ККТ |
|---------|-----------|--------------------------|------------------------------|
| Чеки от ЮKassa | +0,8% от суммы | Нет | Нет |
| Существующая ККТ + ЮKassa | ~0₽ (касса уже оплачена) | Да, ~20ч разработки | Да |
| Существующая ККТ + ПСБ | ~0₽ | Да, ~30ч разработки | Да |

**Вывод:** если существующая ККТ поддерживает API для облачных чеков (большинство современных моделей — да), то при **любом** выборе провайдера можно не платить +0,8%. Нужно уточнить у клиента **модель ККТ**.

---

#### 3.1.3 Добавить раздел «Особенности НКО при подключении к ЮKassa»

Этот раздел отсутствует полностью, но критически важен:

**Перечень документов для ЮKassa (МОО):**
- Устав организации
- Свидетельство о государственной регистрации НКО или выписка из ЕГРЮЛ
- Решение о назначении руководителя (президента)
- Паспорт руководителя
- Реквизиты расчётного счёта

**Сроки:** для НКО проверка занимает 5–10 рабочих дней (против 1–5 для ИП/ООО). Организация зарегистрирована 01.12.2023 — возраст 1,2 года может потребовать дополнительных документов о деятельности.

**Действие:** подавать документы параллельно с началом разработки, не позднее первой недели марта, чтобы к середине марта иметь договор.

---

#### 3.1.4 Уточнить ENV-переменные под оба провайдера

Текущий список содержит только переменные для ЮKassa. Нужно версионировать:

```env
# Активный провайдер
PAYMENT_PROVIDER=yookassa  # или: psb

# ЮKassa
YOOKASSA_SHOP_ID=
YOOKASSA_SECRET_KEY=
YOOKASSA_RETURN_URL=
YOOKASSA_WEBHOOK_URL=
YOOKASSA_IP_WHITELIST=

# ПСБ
PSB_MERCHANT_ID=
PSB_SECRET_KEY=
PSB_API_URL=
PSB_RETURN_URL=
PSB_WEBHOOK_URL=

# ККТ (общие, вне зависимости от провайдера)
KKT_PROVIDER=yookassa_builtin  # или: atol | orangedata | cloudkassir
KKT_API_URL=
KKT_API_KEY=
KKT_GROUP_CODE=
KKT_INN=
OFD_RECEIPT_URL_TEMPLATE=https://receipt.taxcom.ru/v01/show?fn={fn}&fp={fp}&i={fd}
```

### 3.2 Рекомендуемые правки

#### 3.2.1 Добавить раздел «Налоговые особенности» в ТЗ на интеграции

Это не техническая, но операционная документация, которую разработчик должен понимать:

```
Тип платежа          | Чек нужен | Налог УСН 6% | Примечание
---------------------+-----------+---------------+------------------------------------
Вступительный взнос  | НЕТ       | НЕТ           | Целевое поступление, пп. 1 п. 2 ст. 251 НК РФ
Ежегодный взнос      | НЕТ       | НЕТ           | Целевое поступление
Оплата мероприятия   | ДА        | ДА            | Предпринимательская деятельность
```

Это определяет архитектурное решение: **не нужно интегрировать фискализацию для взносов**.

---

#### 3.2.2 Добавить сценарий «Клиент вернулся на сайт до получения вебхука»

Вебхук может опоздать на 5–30 секунд. Пользователь уже на странице `/payment/success`, а статус платежа в БД ещё `pending`. Решение:

```
1. Frontend попадает на /payment/success?payment_id={id}
2. Frontend делает polling: GET /api/v1/payments/{id}/status каждые 2 сек
3. Таймаут: 30 секунд
4. Если за 30 сек статус не изменился → показать «Проверяем оплату... Обновите страницу через минуту»
5. При получении вебхука на бэкенде — статус меняется, следующий poll вернёт succeeded
```

Без этого UX будет сломан для части пользователей.

---

#### 3.2.3 Уточнить Telegram: что делать если бот не может добавить в канал

Текущее ТЗ описывает это как «логировать, уведомить админа». Нужно добавить fallback:

```
1. Попытка добавить через bot.create_chat_invite_link (member_limit=1)
2. Отправить ссылку пользователю в личное сообщение (TTL-ссылка)
3. Если бот не является администратором канала → алерт в TG_ADMIN_CHAT_ID
4. Логировать в monitoring (Sentry)
5. telegram_bindings.is_in_channel оставить false до успешного добавления
```
