## 2. Основные сценарии из документации

### 2.1 СБП QR-коды (C2B — оплата от клиента)

| Этап | Endpoint | Описание |
|------|----------|----------|
| 1 | `get_terminal_parms` | Получение идентификаторов НСПК (SBP_ID, SBP_ACCOUNT_NUMBER, SBP_MERCHANT) |
| 2 | `reg_qr` | Регистрация QR-кода (динамический/статический) |
| 3 | Webhook на `NOTIFY_URL` | Уведомление об успешной оплате |
| 4 | `get_qr_status` | Проверка статуса QR (только для динамических) |
| 5 | `refund` | Возврат |

### 2.2 Подписки (привязка счёта)

- `reg_qr` с параметрами `SUBSCRIPTION_DESC`, `SUBSCRIPTION_NOTIFY_URL` или `SBP_QR_TYPE=SUBSCRIPTION`
- `subscription_payment` — оплата по подписке

### 2.3 B2C — выплаты (юрлицо → физлицо)

- `b2c_reg_payment` — регистрация
- `b2c_confirm_payment` — подтверждение (в течение 180 сек)
- `b2c_cancel_payment` — отмена
- `b2c_check_payment` — статус
- `b2c_check_balance` — баланс счёта

### 2.4 Кассовые ссылки

- `reg_cash_register_qr` — регистрация
- `activate_cash_register_qr` — активация
- `deactivate_cash_register_qr` — деактивация
- `cash_register_qr_status` — статус
- `cash_register_qr_operation_status` — статус операции

### 2.5 Вспомогательные методы

- `get_bank_list` — список банков
- `refund_status` — статус возврата
- `get_qr_data` — данные по QR
- `change_qr_status` — блокировка/разблокировка QR

## 3. Ключевые параметры и подписи

### Подпись (P_SIGN)

Все запросы подписываются. Алгоритм зависит от метода:

| Метод | Порядок параметров для P_SIGN |
|-------|-------------------------------|
| `get_terminal_parms` | стандартные параметры шлюза |
| `reg_qr` | стандартные + SBP_ID, SBP_ACCOUNT_NUMBER, SBP_MERCHANT |
| `get_qr_status` | RND_NUMBER, TERMINAL |
| `refund` | SBP_TRAN_ID, RND_NUMBER |
| `refund_status` | SBP_REFUND_TRAN_ID, RND_NUMBER |
| `refund_get_confirm` | SBP_TRAN_ID, RND_NUMBER |
| `refund_confirm` | SBP_REFUND_TRAN_ID, RND_NUMBER |
| `change_qr_status` | TERMINAL, QR_ID, RND_NUMBER |
| `b2c_cancel_payment` | ORDER, TERMINAL |
| `b2c_check_payment` | ORDER, TERMINAL |
| `b2c_check_balance` | TERMINAL, SBP_ID, SBP_ACCOUNT_NUMBER |

### Тестовые параметры (из документа)

- TERMINAL = `79036768`
- MERCHANT = `790367686219999`
- KEY = `C50E41160302E0F5D6D59F1AA3925C45`

**B2C (выплаты):**

## 6. Flow оплаты для проекта (подписка/мероприятие)

### При использовании ЮKassa

```
Frontend → POST /api/v1/subscriptions/pay
  → Backend создаёт payment
  → Backend вызывает ЮKassa API
  → Response: confirmation_url
  → Frontend redirect на ЮKassa
  → Пользователь оплачивает (карта или СБП)
  → Webhook payment.succeeded → Backend активирует подписку
```

### При использовании прямого ПСБ СБП

```
1. При первом запуске: get_terminal_parms → сохранить SBP_ID, SBP_ACCOUNT_NUMBER, SBP_MERCHANT
2. При создании платежа: reg_qr (AMOUNT, CURRENCY, DESC, NOTIFY_URL, ITN, EMAIL, ...)
3. Показать пользователю QR_IMG_DATA или QR_DATA (ссылку)
4. Ожидать POST на NOTIFY_URL с QR_ID, SBP_TRAN_ID, RESULT=0
5. Если 5 мин нет уведомления — вызвать get_qr_status
6. При успехе: оплата completed, активация подписки/регистрации
```

## 8. Обработка ошибок и edge cases

| Ситуация | Решение |
|----------|---------|
| Уведомление не пришло за 5 мин | Вызвать `get_qr_status` (только для динамических QR) |
| Webhook не ответил 200 | ПСБ повторяет 10 мин, 3 дня |
| REFUND_IF_NOTIFY_FAIL=Y | Банк сам сделает возврат при неудачной доставке |
| Между оплатой и возвратом < 10 мин | Может вернуться «Исходная операция не найдена» — повторить позже |
| Подтверждение B2C | В течение 180 сек, иначе отмена |

5. Сроки внедрения меньше.
